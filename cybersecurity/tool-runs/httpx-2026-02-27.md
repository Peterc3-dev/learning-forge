## 🔷 XRAY (Architecture Analysis)

### Concurrency Model
```
Input (stdin / file / list)
    ↓
Input normalizer (adds http:// and https:// variants)
    ↓
Worker pool (default: 50 concurrent goroutines)
    ↓
Per-host HTTP request → collect response metadata
    ↓
Output writer (thread-safe, buffered)
```

Workers are goroutines pulling from a shared channel. Default concurrency is 50 (`-threads 50`), tunable up or down. Each worker handles one host at a time — HTTP request, response parsing, probe extraction, output.

### What httpx Probes Per Host
| Probe | Flag | What it extracts |
|-------|------|-----------------|
| Status code | `-status-code` / `-sc` | 200, 301, 403, 404, etc. |
| Title | `-title` / `-t` | `<title>` tag content |
| Content length | `-content-length` / `-cl` | Response body size |
| Content type | `-content-type` / `-ct` | MIME type |
| Web server | `-web-server` / `-ws` | Server header (nginx, Apache, etc.) |
| Tech stack | `-tech-detect` / `-td` | Wappalyzer fingerprints |
| Response time | `-response-time` / `-rt` | Latency in ms |
| Line count | `-line-count` / `-lc` | Lines in response body |
| Word count | `-word-count` / `-wc` | Words in response body |
| Hash | `-hash` | MD5/SHA of response — useful for dedup |
| IP | `-ip` | Resolved IP address |
| CNAME | `-cname` | DNS CNAME chain |
| CDN detection | `-cdn` | Identifies Cloudflare, Akamai, Fastly, etc. |
| HTTP/2 | `-http2` | Whether host supports HTTP/2 |
| Follow redirects | `-follow-redirects` / `-fr` | Chase 301/302 chains |
| Screenshots | `-screenshot` | Headless Chrome screenshots (slow, optional) |

### Protocol Handling
httpx automatically tries both HTTP and HTTPS for each input host unless told otherwise. The `-probe` flag makes it report which protocols are alive. This is critical — many subdomains only respond on one protocol.

## 🔷 FIRE (Execution)

### Simplest Command (Baseline)
```bash
echo "example.com" | httpx
```
**Result:** Prints live URLs with status code. If both http and https are up, both appear.

### Standard Recon Command
```bash
httpx -l subdomains.txt -status-code -title -tech-detect -web-server -ip -o live.txt
```
**What you get:** One line per live host with status, title, tech stack, server, and IP. Dead hosts are silently dropped.

### Sample Output (plain)
```
https://api.example.com [200] [API Gateway] [nginx] [1.2.3.4]
https://admin.example.com [403] [Admin Panel] [Apache/2.4] [1.2.3.5]
https://staging.example.com [200] [Staging - Example Co] [React,Node.js] [1.2.3.6]
https://old.example.com [301] [] [nginx] [1.2.3.4]
http://dev.example.com [200] [Development] [PHP/7.4,Laravel] [1.2.3.7]
```

### JSON Output (pipeline-friendly)
```bash
httpx -l subdomains.txt -status-code -title -tech-detect -json -o results.json
```
**Each line is valid JSON:**
```json
{
  "url": "https://api.example.com",
  "status-code": 200,
  "title": "API Gateway",

```
JSON output enables downstream processing with `jq`, feeding into nuclei, sorting by status, etc.

## 🔷 BREAK (Parameter Variation)

### Filter by Status Code
```bash
# Only show 200s (confirmed live, not redirected)
httpx -l subs.txt -status-code -mc 200

# Show 200s and 403s (403 = exists but blocked — worth investigating)
httpx -l subs.txt -status-code -mc 200,403
```
**Why 403 matters:** A 403 means the server is there and actively blocking you. That's a target, not a dead end.

### Rate Limiting for Stealth
```bash
httpx -l subs.txt -rate-limit 10 -timeout 10
```
Slows down to 10 req/sec. Useful when probing targets with WAFs or rate limiting.

### Favicon Hash Matching
```bash
httpx -l subs.txt -favicon
```
Returns the MurmurHash of each favicon. Cross-reference with Shodan (`http.favicon.hash:VALUE`) to find related infrastructure not in your subdomain list.

### Screenshot Mode
```bash
httpx -l subs.txt -screenshot -ss-timeout 15
```
Slow but powerful — headless Chrome renders each page. Output folder contains PNGs. Quick visual triage of hundreds of subdomains.

## 🔷 BRIDGE (Bounty Workflow Integration)

### Where httpx Sits
```
Subdomain Enumeration (Findomain / subfinder)
    ↓
httpx  ← HERE
    ↓ (filters dead hosts, adds metadata)
Live host list with status, tech, server
    ↓
Port scanning (naabu) — on live hosts only
    ↓
Content discovery (ffuf) — on interesting endpoints
    ↓
Vuln scanning (nuclei) — against live.txt
    ↓
Manual investigation of 403s, unusual tech stacks, dev/staging subdomains
```

### High-Value Output Signals
When reviewing httpx output, prioritize:
- **403 responses** — resource exists, access denied. Try bypasses.
- **Dev/staging subdomains** — often less hardened than prod
- **Unusual tech stacks** — old PHP versions, obscure frameworks
- **Large content-length differences** — same URL, different sizes = anomaly
- **Login panels** — look for default creds, auth flaws
- **API subdomains** — `api.`, `v1.`, `internal.` — often underprotected

### Full Pipeline One-Liner
```bash
# Findomain → httpx → nuclei in one shot
findomain -t target.com -q | \
  httpx -silent -status-code -mc 200,403 -o live.txt && \
  nuclei -l live.txt -t ~/nuclei-templates/
```

### Extracting Specific Tech Targets
```bash
# Find all PHP sites (often vulnerable to older CVEs)
httpx -l subs.txt -tech-detect -json | jq -r 'select(.technologies[]? | contains("PHP")) | .url'

# Find all login pages
httpx -l subs.txt -title -json | jq -r 'select(.title | test("login|signin|admin"; "i")) | .url'
```
